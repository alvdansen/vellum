discriminated-union schema. Tool layer remains a thin Zod-validated dispatcher

D-PLAN-4-1: **Single action arm rides in this plan.** Plans 16-03 and 16-04
together complete the three new agent-surface actions (export + verify in
16-03, redact in 16-04). The split aligns with Wave 2/Wave 3 dependency
structure: Plan 16-03 depends only on 16-01 (engine facade for export+verify
already in main); Plan 16-04 depends on 16-02 (redaction engine module).
Splitting 16-03 and 16-04 enables Wave 2 parallelism (16-02 + 16-03 run in
parallel) — putting all three actions in 16-03 would force serial Wave 2
execution.

D-PLAN-4-2: **Envelope mirrors Plan 16-03's export_manifest shape +
redaction-specific fields.** The redact_manifest envelope carries everything
the caller needs to verify the redaction outcome locally:
  - `manifest_bytes_base64` — the freshly-redacted-and-re-signed asset bytes
  - `redacted_fields: string[]` — paths actually redacted (subset of policy)
  - `not_found: string[]` — soft warnings (paths not present in this manifest)
  - `signed_at, format, cert_subject` — verifier-friendly metadata
  - `version_id, breadcrumb, breadcrumb_text` — D-22 / D-PROV-08 envelope
This means a caller can pipe the result of `redact_manifest` directly into
`verify_manifest` (bytes form) for a self-checking round-trip — no separate
state lookup required.

D-PLAN-4-3: **Zod redaction_policy guards are a defence-in-depth duplicate
of the engine's bounded resolver.** The engine's applyRedactionPolicy
(Plan 16-02) rejects policies > 32 entries OR with > 64 segments OR with
regex/traversal characters. The Zod schema at THIS layer adds:
  - `z.array(z.string().min(1).max(1024)).min(1).max(32)`
The character-allowlist enforcement stays at the engine layer (REDACT_POLICY_INVALID
surfaces actionable error). The Zod max(1024) per-entry cap is a coarse
length guard that catches absurd payloads BEFORE the engine validator runs
— prevents an agent from sending a 1 MB-long single-entry policy that would
pass Zod array-length but blow segment-parsing time.

D-PLAN-4-4: **Error code passthrough is verbatim.** The engine emits five
distinct codes (VERSION_NOT_FOUND, REDACT_NO_MANIFEST, REDACT_PARENT_UNREADABLE,
REDACT_POLICY_INVALID, EXPORT_PATH_TRAVERSAL_REJECTED) — the tool layer
preserves each through toolError without rewriting messages. Agents see
the engine's exact code + message + hint; their own retry/fallback logic
keys on the codes verbatim.

D-PLAN-4-5: **Wire-level UAT proves D-CTX-1 invariant.** The integration
test seeds a Phase 14 dev-cert signed manifest with a known sentinel value
embedded (e.g., title='SECRET_TITLE_42'), drives redact_manifest over
both transports with policy targeting the sentinel, and asserts:
  (a) response.structuredContent.redacted_fields contains the policy path
  (b) `Buffer.from(response.structuredContent.manifest_bytes_base64, 'base64')
       .toString('binary').not.toContain('SECRET_TITLE_42')`
The same assertion ran in Plan 16-02's Test 17 at the engine layer; running
it again at the WIRE layer proves the invariant holds across the tool +
transport boundary (no marshaling roundtrip leaks the original).

D-PLAN-4-6 (C-08: envelope vs RedactionResult naming drift): the engine's
`RedactionResult.certSubject` (camelCase, payload-internal) maps verbatim
to the tool envelope's `cert_subject` (snake_case, agent-facing). Phase 14
established `cert_subject_summary` in the manifest_signed payload (the DB
storage shape) — agent-facing envelopes use the shorter `cert_subject` per
Phase 14 D-PROV-08 convention. This naming drift is INTENTIONAL: payload
storage names emphasize "this is a summary, not the full DN" while agent
envelopes use the shorter form (agents already know cert_subject is a
DN summary from PROV-V-07 docs). Document in the source: the ENVELOPE
field is `cert_subject` always; the PAYLOAD field is `cert_subject_summary`
always; the redaction.ts engine surface uses `certSubject` to match its
TypeScript-camelCase neighbors. No fix beyond explicit documentation —
this is a Phase 14-style convention boundary, not a bug.
