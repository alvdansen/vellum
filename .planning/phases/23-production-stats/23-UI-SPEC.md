# Phase 23 — UI Design Contract

> Visual and interaction contract for the sequence-level stats subrow inside `<SequenceHeader/>` (progress bar + backlog callout + stale count) AND the per-shot amber stale border on `<ShotGridCard/>`. Generated by gsd-ui-researcher 2026-05-15 from `.planning/phases/23-production-stats/23-CONTEXT.md` (D-01..D-21 LOCKED, autonomous mode), `.planning/phases/23-production-stats/23-RESEARCH.md` (10 pitfalls + A1-A10 assumptions), `.planning/REQUIREMENTS.md` (OVR-01..OVR-03), `.planning/phases/21-shot-grid-view/21-UI-SPEC.md` (carried-forward design tokens, sequence header structure), `.planning/phases/22-review-and-approval/22-UI-SPEC.md` (QuickApproveButton positioning + popover/overlay coexistence), existing Phase 17-22 design tokens in `packages/dashboard/src/styles/theme.css` (lines 29-120).

**Autonomous mode discretion areas were Claude-resolved.** All four RESEARCH open questions (`--color-stats-backlog-callout` reuse vs new; `stats.total === 0` hide-subrow; EXPLAIN whitelist phrasing; `recomputeIsStaleClient` null branch) are decided inline and annotated `(Claude-selected per autonomous mode)`. Additionally, **A5 WCAG contrast risk surfaced an escalation** — the RESEARCH-recommended `#f59e0b` light-theme stale value FAILS the 3:1 UI-component threshold against `#fafafa` (calculated 2.08:1). This UI-SPEC LOCKS contrast-verified replacement values and documents the deviation from CONTEXT D-08's "recommended" values (the locked values supersede; CONTEXT D-08 explicitly invited UI-SPEC contrast verification).

**Numeric surfaces:** Every count, percentage, and stale number MUST receive class `num` (defined in `theme.css:137-145` as `font-variant-numeric: tabular-nums; font-feature-settings: "tnum"`). This includes:

- ProgressBar numeric value ("60%") — the entire label string SHOULD be `class="num"` because the trailing word " approved" doesn't break tabular alignment of single-element values; conservative reading: wrap just the number in `<span class="num">{pct}</span>` and concatenate the suffix as plain text — RECOMMENDED to avoid surprising vertical alignment if multiple ProgressBars stack.
- Backlog count ("3" in "3 AWAITING REVIEW") — `<span class="num">{n}</span>`
- Stale count ("1" in "1 STALE") — `<span class="num">{n}</span>`

**Uppercase label convention** (LOCKED from v1.0 — `theme.css:148-155`): The backlog callout uses `uppercase tracking-widest` (mirrors `<ShotStatusPill/>` and Phase 21 mini-pills) to match the existing label-pill design vocabulary. The inline stale-count text uses lowercase (matches the timestamp "2h ago" convention in `<ShotGridCard/>`) — it's informational micro-copy, not a pill.

> **Why mixed casing in one row:** The backlog callout is a PILL (the supervisor's primary "do something" signal — uppercase reads as a labeled chip). The stale count is INLINE INFO (the supervisor's secondary "be aware" signal — lowercase reads as ambient text). Treatment difference encodes priority hierarchy. (Claude-selected per autonomous mode.)

### Per-shot stale indicator on `<ShotGridCard/>`

The amber border is a **visual signal only** — no copy is added to the card itself (per D-07 "no separate icon, no badge, no inline 'Stale' text — full-card outline only"). The accessibility cue is delivered via the SR-only aria-label on the existing card button when `is_stale === true`:

| Constant                            | Value                                                                       |
|-------------------------------------|-----------------------------------------------------------------------------|
| `SHOT_CARD_STALE_ARIA_SUFFIX`       | ` — stale`                                                                  |

> Executor appends to the existing `SHOT_CARD_OPEN_ARIA_PREFIX${shot.name}` aria-label on the thumbnail button: when `shot.is_stale === true`, append `SHOT_CARD_STALE_ARIA_SUFFIX` → e.g., `Open version drawer for SH_020 — stale`. This makes the stale state SR-perceivable WITHOUT adding visible text on the card. The border alone is sufficient for sighted users; the aria-suffix is sufficient for SR users.

> **Why no visible text:** Per D-07 the border is the visual signal. Adding visible "stale" text would (a) compete with the QuickApproveButton (Phase 22 D-10 top-right), the ShotStatusPill (D-13 bottom row), and the WarningPill (Phase 22 D-12 inline error) for the card's limited real estate; (b) repeat info that's already conveyed by the border. The aria-suffix is the screen-reader-only equivalent of the border — they're parallel channels, not duplicated info.

### Empty states / hidden states

| Surface                                          | Constant or behavior                                                                       |
|--------------------------------------------------|--------------------------------------------------------------------------------------------|
| Stats subrow when `stats === null` (pre-fetch)   | Subrow does NOT render (returns `null` from component branch). No skeleton. (Claude-selected per autonomous mode — the existing `<SequenceHeader/>` mini-pills row also shows nothing when counts are zero; symmetric.) |
| Stats subrow when `stats !== null && stats.total === 0` | Subrow does NOT render (Open Question 2 resolution — symmetric with Phase 21 D-14 hide-on-zero pattern). The grid-pane empty state "No shots in this sequence yet" already conveys the empty state. |
| ProgressBar when `stats.total === 0` (defensive — subrow hidden anyway) | Component clamps: `Math.max(0, Math.min(100, Math.round(value)))` — never NaN. If subrow somehow renders with zero shots, ProgressBar shows 0% (empty track). |
| Backlog callout when `stats.pending_review_backlog === 0` | Callout does NOT render (returns null from JSX conditional). Same hide-on-zero rule as Phase 21 mini-pills (D-20). |
| Inline stale count when `stats.stale_count === 0` | Text does NOT render (returns null from JSX conditional). Same hide-on-zero rule. |

> **Subrow visibility decision matrix:**
> - `stats === null`: hidden
> - `stats.total === 0`: hidden
> - `stats.total > 0 && stats.approved_pct === 0 && stats.pending_review_backlog === 0 && stats.stale_count === 0`: ProgressBar renders ("0% approved") at minimum; backlog + stale hidden
> - Any non-zero: ProgressBar always renders; backlog renders if > 0; stale renders if > 0

### SSE-driven counter updates (D-10, D-11, D-12 + Pitfalls 1, 2, 3, 8, 9)

The new `sequenceStats: Signal<SequenceStats | null>` is mutated by the EXTENDED `onShotStatusChanged` handler in `state/shot-grid.ts:160`. Algorithm (RESEARCH Pattern 2, with Pitfall hardening):

```
onShotStatusChanged(payload):
  current = shotGrid.value
  if (current === null) return                          // Pitfall 9: pre-fetch SSE dropped (server-snapshot-wins)
  if (current.sequence.id !== payload.sequenceId) return // Pitfall 1: cross-sequence guard

  // Existing per-row update (Phase 21 D-22) — unchanged
  idx = current.shots.findIndex(s => s.id === payload.shotId)
  if (idx >= 0):
    current.shots[idx].status = payload.toStatus

  // NEW Phase 23 — stats delta
  prevStats = sequenceStats.value
  if (prevStats === null) return                        // Pitfall 9 (second guard): stats not seeded yet

  effectiveFromStatus = payload.fromStatus ?? 'wip'     // Pitfall 2: null → 'wip' default
  shotInBuffer = idx >= 0 ? current.shots[idx] : null   // Pitfall 3: outside-buffer = leave stale unchanged

  sequenceStats.value = applyStatsDelta(prevStats, effectiveFromStatus, payload.toStatus, shotInBuffer)
  shotGrid.value = { ...current, shots: [...current.shots] }  // Trigger re-render (immutable replace)
```

`applyStatsDelta` algorithm:

```
applyStatsDelta(prev, fromStatus, toStatus, shotInBuffer):
  newCounts = { ...prev.counts }
  newCounts[fromStatus] = Math.max(0, newCounts[fromStatus] - 1)
  newCounts[toStatus] = newCounts[toStatus] + 1

  newBacklog = newCounts['pending-review']              // Pitfall 10: bracket access for hyphenated key
  newApprovedPct = prev.total === 0 ? 0 : Math.round((newCounts.approved / prev.total) * 100)

  // Stale-count delta (Pitfall 3 + Pitfall 8)
  newStaleCount = prev.stale_count
  if (shotInBuffer !== null):
    wasStale = shotInBuffer.is_stale
    isStaleNow = recomputeIsStaleClient({
      ...shotInBuffer,
      status: toStatus,   // simulate the post-transition state
    })
    if (wasStale && !isStaleNow) newStaleCount = Math.max(0, newStaleCount - 1)
    if (!wasStale && isStaleNow) newStaleCount = newStaleCount + 1
    shotInBuffer.is_stale = isStaleNow   // update the row's flag for the next event

  return {
    total: prev.total,                                    // total never changes via status transitions
    approved_pct: newApprovedPct,
    counts: newCounts,
    pending_review_backlog: newBacklog,
    stale_count: newStaleCount,
  }
```

`recomputeIsStaleClient` algorithm (Open Question 4 resolution — match server exactly):

```
recomputeIsStaleClient(row):
  if (row.latest_completed_version === null) return false  // D-15 grace period — zero versions = not stale
  if (row.status !== 'wip' && row.status !== 'pending-review') return false
  age = Date.now() - row.latest_completed_version.completed_at
  return age > STALE_SHOT_DAYS_MS                         // 14 days * 86_400_000 ms
```

> **Idempotent convergence (Phase 22 D-20 carry-forward):** When the user quick-approves a shot, the optimistic local mutation precedes the PATCH. The PATCH triggers the server-side SSE; the SSE arrives with `toStatus === 'approved'` and the handler runs `applyStatsDelta` — which is idempotent on already-correct state (decrement-from-zero stays at zero via `Math.max(0, ...)`; increment is +1 for one event regardless of whether the local optimistic update already counted it). The shot is keyed on `shotId` and we re-derive `is_stale` from the post-transition state, so the stale-count converges to truth.

> **Multi-tab convergence:** Tab A approves shot X → PATCH → SSE broadcast to ALL tabs. Tab B receives the SSE; if Tab B is viewing the same sequence (`current.sequence.id === payload.sequenceId`), its stats subrow updates. If Tab B is viewing a different sequence, the cross-sequence guard drops the event — stats stay at the previous fetch snapshot (Pitfall 1).

### Initial fetch race (Pitfall 9)

When a user opens a sequence:

1. `ShotGridView` mounts; `fetchShotGrid(seqId)` fires.

3. SSE event for a shot in this sequence arrives → `onShotStatusChanged` runs.

## Edge Cases

### Edge case 1: Sequence has zero shots (`stats.total === 0`)

- **Subrow:** Hidden entirely (Open Question 2 resolution).
- **Grid pane:** Phase 21 empty state "No shots in this sequence yet" renders.
- **Stale border:** No cards to render → no border concern.
- **Backlog callout:** Hidden (Phase 21 D-14 hide-on-zero pattern).

### Edge case 2: All shots are `omit` status

- `stats.total > 0` so the subrow renders.
- `stats.approved_pct === 0` so the ProgressBar shows an empty (0%) track.
- `stats.pending_review_backlog === 0` so the callout is hidden.
- `stats.stale_count === 0` (omit shots are by-def not stale per OVR-02) so the inline stale-count is hidden.
- Grid pane shows the omit shots IF `showOmitted === true` (Phase 21 D-07 toggle); otherwise empty.
- **Net subrow:** ProgressBar (0% approved) only. Reads as "no work to track" — supervisor toggles `Show omitted` to see the omit shots.

### Edge case 3: Sequence with no completed versions (every shot has `latest_completed_version === null`)

- `stats.total > 0` so the subrow renders.
- `stats.stale_count === 0` per D-15 grace period (zero-version shots not marked stale) — inline stale-count hidden.
- `stats.pending_review_backlog` may be > 0 if shots are in pending-review status (manual transition allowed before any version exists, per OVR-02 + Phase 20).
- `stats.approved_pct` is 0 in the typical case (no approved shots without versions).
- **Net subrow:** ProgressBar (0% approved) + optional backlog callout. Reads as "review queue exists but nothing completed yet."

### Edge case 4: Paginated divergence (`stale_count` from server vs. SSE-driven recompute)

- Per D-19 + Pitfall 3: when a SSE event arrives for a shot OUTSIDE the loaded paginated buffer, the client cannot recompute `is_stale` because the shot row isn't in `shotGrid.value.shots`.
- The handler's `shotInBuffer === null` branch leaves `stale_count` unchanged.
- Over a long session with many out-of-buffer transitions, `stale_count` may drift from server truth.
- **Mitigation:** Supervisor refreshes the sequence (home → re-click grid icon) to re-fetch a fresh server snapshot. Documented in `state/shot-grid.ts` JSDoc per RESEARCH Pitfall 3 recommendation.
- **Acceptable for v1.3.** Phase 24+ candidate: 60s background refresh poll if drift becomes a user complaint.

### Edge case 5: Sequence has > 50 shots and stale shot is on page 2+

- Same as Edge case 4 — stale count drifts for out-of-buffer mutations.
- The PER-SHOT stale border still renders correctly when the user loads page 2+ (the server-supplied `is_stale` flag flows through on the fetch).
- Only the AGGREGATE `stale_count` in the subrow may diverge — and only after an SSE-driven transition on an out-of-buffer shot. The border on the actual stale card is always correct on the most recent fetch.

### Edge case 6: Rapid successive SSE events for the same shot

- E.g., shot transitions wip → pending-review → approved within 200ms (test environment or bot agent).
- Each `applyStatsDelta` runs idempotently with the post-transition state.
- The intermediate state (`pending-review`) gets `wasStale = recomputeIsStaleClient` on the wip-row state, then `isStaleNow = recomputeIsStaleClient` on the pending-review-row state.
- The second event uses the row's updated `is_stale` flag (from the first event's `shotInBuffer.is_stale = isStaleNow` write).
- **Correctness:** Counts converge to the final state; `stale_count` reflects the final `is_stale` value.
- **Performance:** Each event is O(1); no batching needed.

### Edge case 7: Restore-from-omit transition (Pitfall 8)

- Shot is omit (omit → not stale per D-15).
- User clicks Restore → status flips to 'wip'.
- `applyStatsDelta` runs with `fromStatus = 'omit'`, `toStatus = 'wip'`.
- `wasStale = shotInBuffer.is_stale = false` (omit shots are not stale).
- `isStaleNow = recomputeIsStaleClient({...shotInBuffer, status: 'wip'})`:
  - If the shot's `latest_completed_version` is null → false (not stale).
  - If the shot's `latest_completed_version.completed_at` is < 14 days ago → false.
  - If the shot's `latest_completed_version.completed_at` is > 14 days ago → **true** (stale).
- In the third case, `stale_count` increments by 1.
- **Correctness verified.** Pitfall 8 documents this exact scenario.

### Edge case 8: Pre-Phase-23 ShotGridView caller passes `stats={undefined}` or omits the prop

- `<SequenceHeader/>` types `stats: SequenceStats | null` (NOT optional `?`).
- If the executor's prop typing is enforced at compile time, callers MUST pass `stats={null}` explicitly.
- If a legacy caller exists (none expected in v1.3 — Phase 21+22 both pass through `ShotGridView` which is updated in this phase), `undefined` is treated as `null` via the `stats === null` falsy check.
- **Defensive:** The component's branch is `{stats !== null && stats.total > 0 && (...)}` — `undefined !== null` is `true`, but `undefined.total` would throw. To be safe, the executor SHOULD code-defensively as `{stats != null && stats.total > 0 && (...)}` (double-equals matches both `null` and `undefined`). RECOMMEND double-equals for this single check.
