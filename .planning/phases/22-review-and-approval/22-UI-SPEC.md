### Focus management

- **Open ReviewPanel:** Focus moves to the action bar's first button (Approve). Executor implements via `useEffect(() => { ref.current?.focus(); }, [shotId])` on the panel mount.
- **Open StatusChangePopover:** Focus moves to the textarea (or to Cancel button if action === 'restore' since textarea is hidden). Mirrors WAI-ARIA dialog convention.
- **Close StatusChangePopover (any path):** Focus returns to the trigger (anchor button). Mirrors `SortDropdown.tsx:147` `triggerRef.current?.focus()` pattern.
- **Open ABCompareView modal:** Focus moves to the modal close button.
- **Close ABCompareView modal:** Focus returns to the "Compare" CTA in the review panel timeline.
- **Status transition succeeds (PATCH 2xx):** Popover closes; focus returns to the anchor action button. The `<ShotStatusPill/>` in the panel header updates in-place (SSE-driven via Phase 21 handler).
- **Status transition fails (PATCH non-2xx or network error):** Popover stays open; an inline error message appears below the textarea ("Failed to change status: {detail} — retry"); the Confirm button re-enables; the user can edit the note and retry or click Cancel.
- **Quick-approve PATCH fails:** `<WarningPill/>` renders inside the card per D-12; auto-dismisses after 5s OR on next successful action on this shot. Focus stays wherever the user moved it after closing the popover (usually nowhere — most users don't tab after a quick-approve click).

### SSE-driven updates (D-20)

- `shot.status_changed` payload `{ shotId, fromStatus, toStatus, changedBy, note? }` continues to flow through the existing `onShotStatusChanged` handler in `state/shot-grid.ts:160-171`. Idempotent — sets `shotGrid.value.shots[idx].status = toStatus` regardless of current value.
- **Optimistic + SSE convergence:** When user quick-approves, local mutation sets `status = 'approved'` BEFORE the PATCH fires. PATCH triggers the SSE on the server side; the SSE arrives with `toStatus = 'approved'` and the handler is a no-op (already-correct value). Multi-tab safety: another tab approving the same shot fires the SSE for all tabs; handler converges all open tabs.
- **In-flight review panel:** Open `ReviewPanel` reads `activeReviewShotId` to determine which shot's history to fetch. SSE-driven updates to the underlying `shotGrid.value.shots` propagate to the panel header (which reads the current shot from `shotGrid.value.shots.find(s => s.id === activeReviewShotId.value)`); the timeline does NOT auto-refresh on every SSE event (would cause scroll jumps); user can close+reopen to re-fetch fresh history if needed. (Claude-selected per autonomous mode — auto-refresh-on-SSE inside the timeline is out of scope; the panel header pill is the primary at-a-glance indicator.)

## URL State Contract

**Phase 22 does NOT mirror review-panel state into the URL.** Review panel open/closed, A/B compare selection, and compare modal visibility are session-only signals.

Justification (Claude-selected per autonomous mode):

- Review panel is fundamentally tied to a specific shot — the `?shot=…` deep link is a Phase 21 concept (already mirrored via `?seq=…&view=shot-grid`); adding `?review=shotId` would let a user share a "review panel open" link, but the SHOT itself is the unit of share-ability (URL already has it).
- A/B compare selection is ephemeral — sharing a "comparing v002 vs v003" link is a Phase 24 candidate; v1.3 ships single-tab session-only.
- The Phase 21 `?seq=…&view=shot-grid&statusFilter=…&showOmitted=…` URL contract is preserved verbatim.

If a future user-facing requirement emerges (e.g., "I want to share a link to a specific A/B comparison"), a follow-up phase can extend the URL shape with `?compare={versionAId}_{versionBId}`.
