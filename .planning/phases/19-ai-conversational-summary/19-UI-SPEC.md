# Phase 19 — UI Design Contract (AI Conversational Summary)

> Visual and interaction contract for Phase 19. Adds a new `<SummarySection/>` block at the top of the version-detail body inside `VersionDrawer.tsx`, REPLACING the existing raw "Provenance" section's primary visibility. Provenance remains available collapsed under a "Show provenance details" disclosure (SUM-07). Adds a `<RegenerateButton/>` (or inline button — planner picks) with 60s server-side throttle countdown + 500ms client-side debounce. Reuses existing `<WarningPill/>` (Phase 12) for the SUM-06 fallback marker and the existing skeleton-shimmer aesthetic for loading state. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope of this contract:**

| Surface | New Component | Existing Component (Modified) | Untouched |
|---------|---------------|-------------------------------|-----------|
| Summary block (top of `VersionDrawer.tsx` body, above `Output` section) | `<SummarySection/>` (or planner may inline JSX in `VersionDrawer.tsx` if reuse-by-other-views is rejected — researcher recommends a separate component for testability + future reuse) | none | `<StatusPill/>`, `<WarningPill/>`, `<C2paBadge/>`, `<JsonBlock/>` (all preserved verbatim) |
| Regenerate affordance (header of summary block) | `<RegenerateButton/>` (NEW; alternative: inline button) | none | All other surfaces |
| Provenance section relocation (SUM-07) | none | `VersionDrawer.tsx` lines 324-337 — wrapped in a `<details><summary>Show provenance details</summary>…</details>` disclosure | The existing `<JsonBlock/>` rendering of each event (preserved verbatim — no API change) |
| Loading state | none — reuses existing `animate-skeleton-shimmer` keyframe + an inline 3-line skeleton block (not the boxy `<SkeletonThumbnail/>`; sentence-shaped skeleton bars) | none | `theme.css` shimmer keyframe (D-WEBUI-26 inheritance) |
| Fallback state | none — composes `<WarningPill/>` (Phase 12, design-token reuse) atop a deterministic-template body text | none | `<WarningPill/>` API surface (props-in-only) |

**Inheritance rule:** every token declared in Phase 5's `theme.css` (spacing scale, color palette, typography, motion timing) and every Phase 17/18 polish-bar pattern (thin-wrapper components, lazy-load contracts, accessibility patterns, lib/copy.ts named-constant copywriting) applies verbatim. This document declares ONLY the deltas Phase 19 introduces. Where a Phase 5 / Phase 17 / Phase 18 token is referenced, it is named (e.g., `--color-surface`, `aria-busy`, `.label-uppercase`) not redefined.

### Hierarchy enforcement rules (Phase 19 additions)

1. **The summary block is the primary content anchor inside the drawer body.** It sits IMMEDIATELY below the drawer header (`<header>` element + the `<section>` for output) — actually FIRST in document order before Output, Timeline, Provenance. The user's eye lands on the summary first when the drawer mounts.
2. **The summary block does NOT have its own surface header chrome beyond the existing `<h3 class="label-uppercase">SUMMARY</h3>` heading.** No badge ("AI"), no decorative icon, no accent border, no shadow. It is a quiet text block; the value is the prose itself.
3. **The Regenerate button sits in the summary block's header row, NOT in the drawer header.** Reason: Regenerate is summary-specific (it regenerates THE SUMMARY, not the version, not the diff). Putting it next to View Diff would conflate two distinct actions. Visual placement: `flex items-center justify-between` so the heading is left, the button is right.
4. **The fallback state DOES NOT degrade the section visually below the success state.** Same surface, same body text color, same heading. The `<WarningPill/>` is a small inline marker (12px height); it acknowledges the fallback without making the section feel broken.
5. **The disclosure for provenance details is COLLAPSED BY DEFAULT.** SUM-07 explicitly says "preserved as table stakes" — collapsed-by-default is the implementation that delivers the contract: provenance is AVAILABLE but not VISIBLE. Users who never expand it never see node JSON; users who need it click once.
6. **No visual decoration on the summary block itself.** No gradient, no inner shadow, no rounded corners on the block beyond the existing `rounded` (4px). Matches Phase 5's "ComfyUI-native" restraint.
7. **The skeleton state (loading) preserves the section's vertical mass.** The 3-line skeleton block at body line-height = 14 × 1.5 + 6px gaps = ~63px ≈ matches the natural mass of a 2-4 sentence Supervisor summary at the same width. CLS=0 when the skeleton swaps to real prose (matches the Phase 17 thumbnail CLS=0 contract; same principle, applied to text).

### Why summary-first, not output-first

The current drawer goes `Output → Timeline → Provenance`. Putting Output first made sense in v1.0/v1.1 because the asset (the thing the user generated) IS the primary value. But v1.2's premise (REQUIREMENTS.md thesis) is that **legibility of the generation context** is the new value tier — VFX Supervisors don't need to look at the dragon's eye to understand what happened; they need a sentence that says "v003 is a tighter close-up of the dragon's eye, generated with Flux + the cinematic_fantasy LoRA at seed 42."

Putting the summary FIRST in the drawer body delivers REQUIREMENTS.md SUM-01 verbatim ("User opens the VersionDrawer and sees a 2-4 sentence Supervisor/Lead-voice summary of the asset and the workflow that made it (instead of a raw node listing)"). The "instead of a raw node listing" is structurally enforced by the SUM-07 disclosure (provenance moved underneath, collapsed).

The Output section remains visible second — the user still sees the actual rendered image without scrolling on a typical 1080p+ drawer height (the summary block at ~120px + header at ~48px leaves ample room above the fold for a typical 360-400px-tall Output thumbnail at the 560px drawer width).

### `<SummarySection/>` API contract (NEW component)

```typescript
import type { VNode } from 'preact';

/**
 * Discriminated state union for the summary block. Mirrors the engine's
 * SummaryOutcome shape from src/engine/summary/index.ts (Phase 19 RESEARCH.md
 * section "Engine Facade — Discriminated Outcome").
 *
 * - 'loading'      — initial fetch in flight (auto-fetch on drawer mount, OR
 *                    user-triggered via Regenerate button)
 * - 'success'      — text is the verified-good LLM output OR a cache hit; the
 *                    UI does not visually distinguish 'live' vs 'cache_hit'
 *                    (researcher discretion: cache-hit-vs-live is server-side
 *                    telemetry, not user-facing — both render identically)
 * - 'fallback'     — text is the deterministic-template summary; <WarningPill/>
 *                    is rendered above the body
 * - 'error'        — fetch failed before any outcome was determined (e.g., 4xx
 *                    INVALID_INPUT envelope on bad versionId, or network error
 *                    on the initial fetch). Renders the SAME fallback UI as
 *                    'fallback' but with a different deterministic-template
 *                    text body (researcher discretion: a single fallback UI
 *                    shape simplifies user mental model — "AI summary
 *                    unavailable" covers both engine-fallback and
 *                    network-error paths). The text body in this case is a
 *                    minimal "(AI summary unavailable; please retry.)" string
 *                    since the engine never produced a deterministic template
 *                    for this fetch.
 */
export type SummaryState =
  | { state: 'loading' }
  | {
      state: 'success';
      text: string;
      // Server-side telemetry; the dashboard does NOT render these visibly.
      // Kept on the type so the dev-tooling / e2e tests can assert on them.
      source: 'live' | 'cache_hit';
      generated_at: string; // ISO-8601 from server
      template_version: string; // semver string from server (e.g., '1.0.0')
      model_id: string; // e.g., 'claude-haiku-4-5-20251001'
    }
  | {
      state: 'fallback';
      text: string; // deterministic-template content from the engine
      // Server-side telemetry; not rendered visibly.
      source: 'fallback';
      // Optional reason code from the engine (research RESEARCH.md fallback
      // triggers: 'circuit_open' | 'api_key_missing' | 'validation_failed' |
      // 'network_after_retry'). UI does NOT branch on this — single fallback
      // visual.
      reason?: string;
    }
  | {
      state: 'error';
      // No text body — UI renders the minimal fallback string from copy.ts.
      // Kept on the type so the test harness can assert on the fetch error.
      message?: string;
    };

export interface SummarySectionProps {
  /**
   * The current summary state for the version this drawer is rendering.
   * Parent (`VersionDrawer.tsx`) reads this from `summarySignal[version.id]`
   * (or threads it down explicitly via prop — planner picks signal-read vs
   * prop-drilling based on parent ergonomics).
```

**Render contract:**

| State | Label | Disabled? | aria-busy |
|-------|-------|-----------|-----------|
| Default (no cooldown, no fetch in flight) | `Regenerate` | false | false |
| Cooldown active (`cooldownSeconds > 0` and not fetching) | `Regenerate ({cooldownSeconds}s)` with `.num` / `tabular-nums` on the `({cooldownSeconds}s)` span | true | false |

**Cooldown countdown implementation:**

```tsx
// Inside RegenerateButton.tsx — owns its own timer for the 1Hz countdown tick
const [now, setNow] = useState(() => Date.now());
useEffect(() => {
  if (!regenerateAvailableAtMs || regenerateAvailableAtMs <= now) return;
  const id = setInterval(() => setNow(Date.now()), 1000);
  return () => clearInterval(id);
}, [regenerateAvailableAtMs, now]);
const cooldownSeconds = regenerateAvailableAtMs && regenerateAvailableAtMs > now
  ? Math.ceil((regenerateAvailableAtMs - now) / 1000)
  : 0;
```

The 1Hz tick is acceptable per `prefers-reduced-motion: reduce` because changing TEXT (not animation) does not violate the reduced-motion contract. The button's disabled state + countdown serves users who would otherwise spam-click.

## Interaction Contract

### Summary block content auto-fetch (D-WEBUI-31 / Phase 14 precedent)

The summary auto-fetches on `VersionDrawer` mount via a `useEffect([version.id])` hook — mirrors the Phase 14 C2PA-status auto-fetch at `VersionDrawer.tsx:119-133` verbatim. Behavior:

| Trigger | Behavior |
|---------|----------|
| Drawer mounts with a new `version.id` | `setSummary({ state: 'loading' })` → call `getSummary(versionId)` → on response, `setSummary(<server response mapped to SummaryState>)` → on error, `setSummary({ state: 'error' })` |
| Drawer remains mounted, `version.id` changes (user clicks different VersionCard) | Same as mount — the effect re-runs because `version.id` is in the deps array |
| Drawer remains mounted, `version.id` does NOT change | Effect does NOT re-run — the existing summary stays visible |
| Drawer unmounts mid-fetch | `let alive = true` guard at the top of the effect; `if (alive) setSummary(...)` inside `.then`/`.catch`; cleanup function sets `alive = false`. Identical pattern to existing C2paStatus effect. |

The fetch helper `getSummary(versionId)` lives in `packages/dashboard/src/lib/api.ts` (NEW function — alongside `getC2paStatus` from Phase 14, `getThumbnailUrl` from Phase 17, `fetchVersions` from Phase 18). Returns a discriminated union mapping the server's `{ text, source, generated_at, template_version, model_id }` envelope to the `SummaryState` shape. NEVER throws — defence in depth: network errors, missing fields, malformed JSON all collapse to `{ state: 'error' }`.

### Regenerate button click handling

| Step | Behavior |
|------|----------|
| 1. User clicks Regenerate | If button is `disabled`, no-op (HTML native). Otherwise, the click handler fires. |
| 2. 500ms client-side debounce | The handler captures the click in a `useRef<number>` timestamp; if a previous click was within 500ms, it's a no-op. (CONTEXT.md SUM-04 / D-FB-4: "500ms debounce client-side"). |
| 3. Optimistic UI flip | `setSummary({ state: 'loading' })` — body swaps to skeleton, button label becomes "Regenerating…". |
| 4. Server fetch | Call `regenerateSummary(versionId)` (NEW helper in `lib/api.ts`) → POST `/api/versions/:id/summary/regenerate`. |
| 5. Server response | (a) 200 OK with new envelope → map to `SummaryState` and `setSummary(...)`. (b) 429 Too Many Requests with `regenerate_available_at_ms` → revert to previous summary state AND set the cooldown timer. (c) 5xx with deterministic-template fallback envelope → `setSummary({ state: 'fallback', ... })`. (d) Network error → `setSummary({ state: 'error' })`. |
| 6. Cooldown timer starts | The new envelope (whether success or fallback) carries `regenerate_available_at_ms = now + 60000`; the dashboard reads this and starts the 1Hz countdown. |
| 7. Button re-enables when cooldown expires | The `RegenerateButton` internal `setInterval` decrements the displayed cooldown; when `cooldownSeconds === 0`, the button label flips back to "Regenerate" and `disabled` is removed. |

### Disclosure (Show provenance details) click handling

Native `<details>` / `<summary>` — no custom click handler. The browser toggles the `[open]` attribute on click; the disclosure body (the existing `<ul>` of `<JsonBlock/>` records) renders/unrenders accordingly. NO state lift to React/Preact; native HTML manages the toggle. NO localStorage persistence (researcher discretion: per-version disclosure state is not load-bearing UX; user expectation matches "fresh drawer mount = collapsed"; persistent expanded-state would clutter local storage and require key-per-version eviction — defer to v1.3+ if user feedback signals demand).

### Performance contract

- Summary auto-fetch on drawer mount adds 1 HTTP round-trip per drawer-open. The 60s server-side throttle bounds regenerate frequency. Cache hits (manifest_sha256 + template_version + model_id match) return in < 50ms (single SQLite SELECT) — no LLM latency on cached views. Cache misses (live LLM call) return in ~600ms-12s (1s retry + 10s timeout cap per RESEARCH.md). The 3-line skeleton bridges the loading window.
- Regenerate explicitly bypasses the cache; expected latency ~600ms-12s for a fresh Haiku 4.5 call.
- The cooldown timer's 1Hz `setInterval` is a single timer per mounted RegenerateButton component. Negligible CPU cost. Cleanup on unmount via the `useEffect` return function.
- NO blocking paint on summary fetch — the drawer renders immediately with the loading skeleton, then transitions to real prose when the fetch resolves. Matches Phase 14 C2paBadge auto-fetch lazy-render pattern.
