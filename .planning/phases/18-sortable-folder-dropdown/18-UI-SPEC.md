# Phase 18 — UI Design Contract (Sortable Folder Dropdown)

> Visual and interaction contract for Phase 18. Adds the new `<SortDropdown/>` component, a "Load more" pagination button, and the visible sort-control strips above the version grid (right pane) and the TreeSidebar (left pane). Inherits the dashboard's locked design system from Phase 5 (live in `packages/dashboard/src/styles/theme.css`) and Phase 17's thin-wrapper polish bar (`<Thumbnail/>` / `<C2paShield/>` / `<WarningPill/>` / `<StatusPill/>`). Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope of this contract:**

| Surface | New Component | Existing Component (Modified) | Untouched |
|---------|---------------|-------------------------------|-----------|
| Version grid header strip (right pane in `HomeView.tsx`) | `<SortDropdown/>` (grid instance, 4 options) | `HomeView.tsx` `<main>` block — wraps current `<ul>` in a vertically-stacked layout with a header strip + grid + "Load more" button | `<VersionCard/>` (untouched), `<EmptyState/>` (untouched) |
| TreeSidebar header strip (left pane in `HomeView.tsx`) | `<SortDropdown/>` (tree instance, 4 options) | `HomeView.tsx` left-pane composition — wraps current `<TreeSidebar/>` in a vertically-stacked container with a header strip above | `<TreeSidebar/>` itself stays a pure pass-through-props component (D-06 — header sits ABOVE the `<nav>` in `HomeView.tsx`, not inside `TreeSidebar.tsx`) |
| Pagination affordance (version grid bottom) | `<LoadMoreButton/>` (or inline button — planner picks based on reuse signals) | `HomeView.tsx` `<main>` block | All other surfaces |
| In-progress band visual (top-pinned) | none — reuses `<VersionCard/>` running/queued render path which already shows `<SkeletonThumbnail/>` via `<Thumbnail/>` (D-04) | none | `<SkeletonThumbnail/>` (D-04 LOCKED — reused as-is) |

**Inheritance rule:** every token declared in Phase 5's `theme.css` (spacing scale, color palette, typography, motion timing) and every Phase 17 polish-bar pattern (thin-wrapper components, lazy-load contracts, accessibility patterns) applies verbatim. This document declares ONLY the deltas Phase 18 introduces. Where a Phase 5 / Phase 17 token is referenced, it is named (e.g., `--color-surface`, `aria-busy`) not redefined.

### Hierarchy enforcement rules (Phase 18 additions)

1. **The sort-strip is visually subordinate to the content below it.** Smaller height (40px vs the grid's per-card ~96px), muted "Sort:" label, compact dropdown trigger. The user's eye lands on the grid first, the strip second.
2. **The dropdown popover overlays the grid; it does NOT push content down.** Use `position: absolute` (anchored to the trigger) so opening the dropdown doesn't shift the version cards below. Avoid CLS — same rule as Phase 17 thumbnails.
3. **The "Load more" button is visually subordinate to the cards above it.** Surface tone (no accent fill — surface fill); muted secondary count text; smaller text size than the cards' label text.
4. **No visual decoration on the strip itself.** No gradient, no inner shadow, no rounded corners on the strip container. The strip is a thin horizontal band with a single bottom border. Consistent with Phase 5's restrained motion and the dashboard's "ComfyUI-native" tone.
5. **The pinned in-progress band has NO new visual chrome.** No "In-progress" header label, no background tint, no separator line between the pinned band and the rest of the sorted list. The composite cursor + NULLS-FIRST ordering is structurally invisible to the user — they see a unified vertical list where in-progress cards happen to appear at the top first (D-04).

### `<SortDropdown/>` API contract (NEW component)

```typescript
import type { VNode } from 'preact';

/**
 * One option in the dropdown. Encodes both the sort field and direction so
 * the dropdown is a single-select over (field × dir) pairs — NOT a separate
 * field-picker + direction-toggle. Phase 18 ships 4 options per pane with
 * the field+dir pair encoded in the option `id`.
 */
export interface SortOption {
  /**
   * Stable identifier used as the encoded value in localStorage and URL state.
   * Format: `${field}:${dir}` (e.g., 'completed_at:desc', 'name:asc'). Validated
   * against the engine's whitelist enum at the URL parse boundary AND at the
   * localStorage read boundary (D-16, D-24). Invalid id → fallback to default.
   */
  id: string;
  /** User-facing label (e.g., 'Latest', 'Name A→Z'). Plain text — no HTML. */
  label: string;
}

export interface SortDropdownProps {
  /**
   * The full list of options visible in the popover. Caller passes the
   * appropriate set:
   *   - Grid: 4 options [Latest, Oldest, Name A→Z, Version ↓]
   *   - Tree: 4 options [A→Z, Z→A, Newest, Oldest]
   * The component does not validate the ids — caller's responsibility (the
   * engine ORDER BY whitelist and the URL/localStorage validators are the
   * authoritative gates per D-16 and D-24).
   */
  options: SortOption[];
  /**
   * The currently-selected option id. Must match one of `options[i].id`.
   * If no match, the trigger renders the first option's label (defensive
   * fallback — the parent should have already coerced an invalid value to
   * the default before passing it down).
   */
  value: string;
  /**
   * Fired when the user picks an option. Caller persists the new value to
   * localStorage (`vfx-familiar:sort:grid` or `vfx-familiar:sort:tree`) and
   * mirrors to URL state (`?gridSort=…&treeSort=…`) via `history.replaceState`.
   * The component itself does NOT touch localStorage or window.history.
   */
  onChange: (id: string) => void;
  /**
   * ARIA label for the trigger button. Required (D-discretion):
   *   - Grid instance: 'Sort versions by'
   *   - Tree instance: 'Sort tree by'
   */
  ariaLabel: string;
  /** Optional class for the outermost wrapper (composition with parent). */
  class?: string;
}

export function SortDropdown(props: SortDropdownProps): VNode { /* ... */ }
```

**Render contract:**

| State | Trigger render | Popover render |
|-------|----------------|----------------|
| Closed (default) | `<button aria-haspopup="listbox" aria-expanded="false" aria-label={ariaLabel}>` containing the current option's label + `<ChevronDown size={14}/>` (rotation 0deg) | NOT rendered (no DOM nodes for the popover when closed) |
| Open | Same trigger; `aria-expanded="true"`; chevron rotates 180deg via Tailwind `rotate-180` | `<ul role="listbox" aria-label={ariaLabel}>` absolutely positioned below the trigger; one `<li role="option">` per option |
| Option selected | trigger label updates synchronously; popover closes | popover unmounts; `aria-activedescendant` cleared |

**Interaction contract:**

| Trigger | Behavior |
|---------|----------|
| Click trigger button | Toggle open/closed |
| Click option | Fire `onChange(option.id)`, close popover, restore focus to trigger |
| Click outside popover | Close popover, restore focus to trigger |
| `Escape` key | Close popover, restore focus to trigger |
| `Enter` / `Space` on trigger (focused) | Toggle open/closed |
| `ArrowDown` on trigger (focused, closed) | Open popover, focus first option |
| `ArrowUp` / `ArrowDown` inside popover | Move keyboard focus through options (wraps at boundaries — `ArrowDown` on last option → first option) |
| `Enter` / `Space` on focused option | Select option (same as click) |
| `Home` inside popover | Focus first option |
| `End` inside popover | Focus last option |
| `Tab` inside popover | Close popover, advance focus to next focusable element after trigger |

**Accessibility contract (combobox-like pattern, simplified for select-only — NO type-ahead in v1.2):**

| Property | Value | Why |
|----------|-------|-----|
| Trigger element | `<button type="button">` | Native button gives keyboard activation + screen-reader semantics for free |
| Trigger `aria-haspopup` | `"listbox"` | WAI-ARIA combobox 1.2 pattern for select-only dropdowns |
| Trigger `aria-expanded` | `"true"` when open, `"false"` when closed | SR users hear "expanded" / "collapsed" state |
| Trigger `aria-label` | From prop (`ariaLabel`) | Required (no visible label inside the trigger besides the option text) |
| Trigger `aria-controls` | id of the popover `<ul>` when open | Wires the trigger to its controlled list |
| Popover `<ul>` `role` | `"listbox"` | WAI-ARIA listbox pattern |
| Popover `<ul>` `aria-label` | Same as trigger | SR users hear the listbox label when arrow-keying into it |
| Option `<li>` `role` | `"option"` | WAI-ARIA listbox option |
| Option `<li>` `aria-selected` | `"true"` for currently-selected option, `"false"` for others | SR users hear which option is the current sort |

```typescript

  /**
```

| State | Render |
|-------|--------|

## Interaction Contract

### Sort change flow (D-12 / D-13 / D-14 / D-15 LOCKED)

When the user clicks a sort option in either dropdown:

1. **Dropdown closes**, focus restored to trigger.
2. **`onChange(option.id)` fires** in the parent (`HomeView.tsx`).
3. **Parent updates the signal** (`gridSort.value = option.id` or `treeSort.value = option.id`).
4. **Parent writes to localStorage** (`vfx-familiar:sort:grid` or `vfx-familiar:sort:tree`) using the bounded-key LRU helper from D-25.
5. **Parent updates the URL** via `history.replaceState({}, '', '?gridSort=…&treeSort=…')` — both pane parameters always present per D-15.
6. **For grid sort change:**
   a. Cursor reset to null (D-19 — page 1).
   b. New `fetchVersions(shotId, { sort, cursor: null, limit: 20 })` fires.
   c. `<main>` `<div class="overflow-y-auto">` `scrollTop = 0` (D-19).
   d. While in flight: existing UX (signal momentarily empty → `<EmptyState/>` shows briefly).
7. **For tree sort change:**
   a. Client-side re-sort of the cached `children.projects` / `children.sequences` / `children.shots` arrays per the new sort key (D-discretion: "Recommended: client-side re-sort").
   b. Already-loaded children re-render in the new order; no fetch fires.
   c. Future hierarchy fetches (lazy-loaded children for not-yet-expanded nodes) include the `sort` parameter.
8. **No animation on the sort transition** — the cards re-order instantly. NO list-flip animation (FLIP-style). Phase 18 ships with the conservative instant re-render.

### Pagination flow ("Load more" — D-17 / D-18 / D-20 LOCKED)

When the user clicks "Load more":

1. **Button enters loading state** (`isFetching = true`, label swap to "Loading…", `aria-busy="true"`, `disabled` attribute set).
2. **`fetchVersions(shotId, { sort, cursor, limit: 20 })` fires** with the current cursor.
3. **On success:** new items append to the `versions` signal (the signal becomes a paginated buffer — D-21). `next_cursor` updates. `isFetching = false`. The pinned in-progress band is page-1-only (D-20) — subsequent pages contain only completed cards.
4. **On error:** `isFetching = false`; `loadMoreError = <error message>`. Inline error pill appears below the button. Button itself stays clickable (Retry == re-clicking the button).
5. **On `next_cursor === null`:** `hasMore = false`; the parent unmounts the `<LoadMoreButton/>` component (renders nothing).

### URL precedence on first load (D-13)

On dashboard mount:
1. **Read URL** for `?gridSort=…&treeSort=…`. Validate against the engine ORDER BY whitelist.
2. **If URL has a valid value:** use it. localStorage stays UNTOUCHED (D-13).
3. **If URL has an invalid value (D-16):** fall back to default; replaceState the URL with the default; `console.warn` the malformed value.
4. **If URL is absent:** read localStorage. Validate same whitelist.
5. **If localStorage has a valid value:** use it. URL updates via `replaceState` to reflect the localStorage value (D-15 — URL always shows current sort).
6. **If localStorage has an invalid value (D-24):** fall back to default; overwrite localStorage with the default; `console.warn`.
7. **If localStorage is absent:** use the default (`completed_at:desc` for grid, `name:asc` for tree). URL updates via `replaceState` to reflect the default.

After mount, sort changes always write to BOTH localStorage AND URL (per D-13 closing clause).

### Performance contract

- **Sort change fetch:** the signal becomes `[]` momentarily during refetch. The flicker is acceptable for a v1.2 conservative ship (no skeleton-list state added — would be over-design for a 100-300ms fetch).
- **"Load more" fetch:** the existing cards stay rendered (the signal appends rather than replaces). Only the button enters a loading state. CLS = 0 on the appended cards because `<Thumbnail/>` declares explicit `width`/`height` (Phase 17 inheritance).
- **Tree client-side re-sort:** the cached arrays are re-sorted in JavaScript using `Array.prototype.sort` with a comparator built from the sort key. For typical project sizes (≤ 100 nodes per level), this is sub-1ms — no perceptible delay. NO virtualized list needed.
- **localStorage I/O:** synchronous reads on mount (cheap — ≤ 1ms); writes on sort change (cheap — ≤ 1ms). Wrapped in try/catch (existing `ThemeToggle.tsx` pattern).
- **URL update:** `history.replaceState` is sub-1ms; no fetch implications.

Mitigation: planner can add `right-0` Tailwind utility to anchor the popover to the trigger's right edge (instead of left edge) when the trigger's right-edge X position is within `popover_min_width` of the viewport right edge. v1.2 conservative ship: anchor below-left always (the dashboard's minimum 1024px width gives ≥ 884px main pane even with the 280px sidebar; a 180px popover anchored below-left from any trigger position has ample room).
