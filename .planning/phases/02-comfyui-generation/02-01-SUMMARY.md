## Decisions Made

- **Drizzle migration sequencing:** The plan's `<action>` block anticipated that `drizzle-kit generate` would produce a 0000 baseline that re-creates the tables (Phase 1 SCHEMA_DDL already creates them via `CREATE TABLE IF NOT EXISTS`). Chose the cleaner of the two strategies in the plan's fallback guidance: delete the baseline 0000 SQL, rename the snapshot to `0001_snapshot.json`, and manually craft `_journal.json` to list only the `0001_phase2_version_lifecycle` entry. Verified at runtime by reading `drizzle-orm/migrator.js` — the runtime migrator only reads journal entries and their matching `.sql` files; snapshots are a `drizzle-kit generate`-time artefact and have no runtime effect. This gives fresh DBs (SCHEMA_DDL creates Phase 1 tables) and existing DBs (tables already present) the same migration code path: `ALTER TABLE` adds the three new columns, `__drizzle_migrations` records a single applied row.
- **Drizzle transaction API:** The research `<action>` template showed `return this.db.transaction((tx) => { ... })();` with a trailing call-parens. The actual `drizzle-orm/sqlite-core` signature (`transaction<T>(cb: ... => Result<...T>): Result<...T>`) returns the callback result directly. The trailing `()` caused `TypeError: ... is not a function`. Fixed by removing the trailing `()` during the TDD iteration; all 11 VersionRepo tests passed on the next run.
- **isUniqueViolation duplication:** Kept a local copy in `version-repo.ts` (identical body to `hierarchy-repo.ts`) per PATTERNS.md callout. Alternative would have been extracting to a shared helper, but that introduces cross-repo coupling the plan explicitly declined.
- **SCHEMA_DDL left unchanged:** Per plan instruction, only the Drizzle table definition was updated with the three new columns. `SCHEMA_DDL` continues to produce Phase 1 shape on fresh DBs; the migration applies on top. This means the pragma-first bootstrap (`user_version=0` check) and the Drizzle migrator are two different mechanisms that operate at different layers — documented in `drizzle.config.ts` comments.

## VersionRepo state-machine behavior (one paragraph)

`VersionRepo.insertVersion(shotId, notes?)` runs inside `this.db.transaction((tx) => { ... })` — SQLite's BEGIN IMMEDIATE semantics under better-sqlite3. Inside the transaction, a `SELECT COALESCE(MAX(version_number), 0) + 1` reserves the next integer and an `INSERT INTO versions ...` adds the row at status `submitted`. On `UNIQUE(shot_id, version_number)` violation (rare concurrent-submit race), the outer loop retries once, then throws `TypedError('CONCURRENT_SUBMIT_CONFLICT', ...)`. Terminal updates use raw `sql` template tags with the guard clause `WHERE id = ${id} AND completed_at IS NULL` — `markCompleted(id, outputsJson)` and `markFailed(id, code, message)` both enforce D-GEN-20 immutability (second call becomes a no-op). `transition(id, 'running')` is the only non-terminal state advance; `listPendingVersions()` returns rows where `status IN ('submitted', 'running')` for the recovery poller.
