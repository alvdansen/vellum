<objective>
Wire Plan 16-01's engine facade methods (`Engine.exportManifestForVersion` +
`Engine.verifyManifestForVersion`) through the version tool's discriminated-
union schema, adding TWO new action arms: `export_manifest` and
`verify_manifest`. Tool layer is a thin Zod-validated dispatcher (D-33 /
architecture-purity) — zero engine logic inline; zero c2pa-node imports.

D-PLAN-3-1: **Two action arms ride in this plan.** Plan 16-04 adds the third
arm (`redact_manifest`). Splitting export+verify into separate plans would
double the test fixtures (each transport needs an independent server-spawn
harness) without buying parallelism — they share the engine-facade contract
from Plan 16-01 and the same wire-level UAT scaffolding. CONTEXT.md specifics
block proposed this exact bundling.

D-PLAN-3-2: **verify_manifest input is a Zod union of two discriminated
shapes.** Per CONTEXT.md D-CTX-4: a single `action: 'verify_manifest'`
literal with EITHER `{version_id}` OR `{manifest_bytes_base64, format}`. The
discriminator is the FIELD presence, not a separate enum — Zod's
`z.union([...])` resolves this at parse time. Cleaner than two distinct
action literals because the engine facade already accepts the same union
shape.

D-PLAN-3-3: **Envelope shape locked to D-PROV-08 dual-form.** structuredContent
+ content[0].text JSON-string mirror. New fields (manifest_bytes_base64,
matched_assertions, gaps, failures, cert_subject, signed_at,
ingredients_summary) join the existing breadcrumb + breadcrumb_text per
D-22. No new envelope mechanism.

D-PLAN-3-4: **Breadcrumb is NULL for pure-bytes verify path.** When the
caller invokes `{action: 'verify_manifest', manifest_bytes_base64, format}`
without a version_id, the engine has nothing to resolve a breadcrumb from
— the bytes could come from anywhere. Setting breadcrumb=null in the
envelope is the honest signal. Plan 16-04's redact_manifest action ALWAYS
has a version_id and always carries a breadcrumb.

D-PLAN-3-5: **Payload-size guard for verify-by-bytes.** T-16-05 (Plan 16-01
threat model) accepted v1.1's base64-in-envelope shape. The reverse channel
— bytes flowing INTO the tool from the agent — is now exposed via this
plan's verify_manifest action. We add a length-check at the tool boundary
(~700 MB base64 cap, mirror of BUFFER_SIGNING_MAX_BYTES_BASE64) so a
malicious or accidental oversized payload returns INVALID_INPUT
immediately, before the engine attempts the c2pa.read.

Purpose: This plan is Wave 2, parallel with Plan 16-02 (redaction engine)
because their files_modified sets do NOT overlap. Plan 16-04 (redact_manifest
tool action) is Wave 3 — it depends on 16-02 (redaction engine module). Plan
16-05 (E2E + cohort closure) is Wave 4 — depends on 16-03 + 16-04.

Output: Two new tool actions wired through the engine facade + comprehensive
unit + dual-transport integration tests. PROV-V-07 substantially complete
at the wire boundary (cohort closure in Plan 16-05).
</objective>
